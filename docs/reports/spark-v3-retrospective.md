## 1. What We Set Out to Do

SPARK v3 was the third major iteration of workgraph's self-organizing development cycle. The goal: make workgraph capable of managing its own development autonomously -- dispatching tasks, evaluating results, evolving its own prompts, and recovering from failures without human intervention.

### Planned Tracks

| Track | Scope |
|-------|-------|
| **Safety** | Fail/retry/abandon commands, service guard, zero-output detection, dependency enforcement, mandatory validation |
| **Infrastructure** | TOCTOU race fixes, liveness detection, compactor (context.md generation), atomic task mutations, per-role model routing |
| **Agency** | Evaluation pipeline, FLIP (Faithful Likelihood of Implementation Precision), evolver (prompt self-improvement), cost tracking, executor weight tiers |
| **Human** | Notification channels (Telegram, webhook), NotificationChannel trait, human-in-the-loop approval flows |
| **TUI** | Inspector views, firehose, service health badge, token display, markdown rendering, graph visualization, touch support, message input |
| **Lifecycle** | Task state machine, cascade abandon, retry+re-eval, supersession, zombie detection, convergence |
| **Observability** | Agent activity dashboard, stream events, lifecycle indicators, FLIP pipeline visibility |

### Original Scope

The task graph contained 335 total tasks (including system-generated .assign, .evaluate, and .verify-flip tasks). Of these, 108 were user-defined implementation tasks covering the tracks above, with 227 system tasks auto-generated by the coordinator for assignment, evaluation, and FLIP verification.

## 4. Process Learnings

### What Worked

**Fan-out/fan-in pattern:**
The most productive pattern was research -> design -> implementation with parallel fan-out. Example: agent isolation went through research (GitButler, git worktrees) -> synthesis (decision document) -> parallel implementation (spawn, CWD, lifecycle cleanup). Each phase produced artifacts that the next phase consumed.

**FLIP verification pipeline:**
Having an independent second opinion on task quality caught genuinely low-quality work. Tasks with FLIP < 0.70 triggered verification, and several were rejected and restarted. This created a quality feedback loop without human oversight.

**Research -> design -> implement pattern:**
42 research documents and 41 design documents were produced. Having agents research before implementing led to better decisions (e.g., choosing worktrees over GitButler, choosing flock over advisory locks).

**Evaluation-driven evolution:**
The evolver's data-driven approach -- citing specific score data to justify mutations -- produced meaningful improvements. Retiring redundant variants (Programmer-Usable, Documenter-StyleAware) kept the role space manageable.

**Conventional commit discipline:**
47.7% of commits include task ID tracing, enabling post-hoc analysis of which tasks produced which code. Strong commit message conventions (53.5% feat, 23.9% fix) make the git log readable.

### What Didn't Work

**Shared working tree conflicts:**
The #1 process failure. All agents operating in a single git worktree led to 38 stashes, cross-agent contamination, and lost work. Root cause: `spawn_agent_inner()` doesn't isolate agent file systems. Workaround: worktree isolation was designed and implemented mid-cycle, but the historical damage (stashes, unmerged branches) remains.

**Zombie agents and eval timeouts:**
Agents that produced zero output consumed spawn slots and required circuit-breaker intervention. The zero-output detector was built reactively to address this problem. Evaluation timeouts created noise in the eval pipeline.

**Unmerged branches:**
Only 2 of 9 feature branches were ever merged to main. Tasks are marked "done" when committed to a feature branch, but code doesn't reach main. This means the current state depends on which branch you're on.

**Human integration quality:**
The notification channel track produced the lowest FLIP scores (0.29-0.46). The code compiles and has structure, but there's no evidence it actually works end-to-end with real Telegram/webhook endpoints.

**Evolver auto-trigger gap:**
The evolver ran manually (via `wg evolve`) rather than auto-triggering from the coordinator. The auto-trigger infrastructure exists in code (thresholds, reactive triggers, budget caps) but `evolver_state.json` was never created. The loop from observation to self-improvement doesn't close automatically.

### How the System Self-Organized

The coordinator's 7-phase tick loop drove all automation:
1. Dead agent cleanup
2. Zero-output sweep
3. Auto-evaluate completed tasks
4. FLIP scoring
5. Spawn new agents on ready tasks
6. Compactor check
7. Evolution check (never triggered)

Phases 1-5 ran continuously. The system successfully self-organized task execution: creating .assign tasks, dispatching agents, evaluating results, and generating FLIP verification for low scores. This happened 323 times without human intervention.

Human intervention was needed for:
- Setting strategic direction (which tasks to create)
- Branch management (merging, resolving divergence)
- Running evolution manually (`wg evolve`)
- Monitoring for systemic issues (watching agent churn, checking stash accumulation)

## Summary

SPARK v3 accomplished its primary goal: workgraph can dispatch its own tasks, evaluate results, and recover from failures without human intervention. The system completed 323 of 335 tasks (96.4%) across 7 tracks with a mean evaluation score of 0.803 over 1,207 evaluations.

The key gap is the learning loop: the system observes its own performance (via evals and FLIP) but doesn't yet autonomously apply those observations to improve (evolver auto-trigger inactive, evolved amendments not consumed). Closing this loop would move workgraph from "self-sustaining for execution" to "genuinely self-improving."

The other major debt items are git hygiene (38 stashes, 7 unmerged branches) and documentation staleness. Both are tractable problems with clear solutions.

**Autopoietic score: 3.5/5** -- functional and largely self-sustaining, with specific gaps that prevent a higher rating.
